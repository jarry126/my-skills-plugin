# 执行前澄清

## 自动触发

任务落入 frontmatter 描述的适用范围时直接使用，不要求用户输入 `clarify-before-execute`、`使用该 skill` 或其他固定口令。是否追问取决于关键不确定性和返工风险，而不是是否出现 skill 名称。

## 核心目标

在执行任何任务前，先判断信息是否足够。不要在关键条件不清楚时直接动手。

这里的“95% 信心”不是数学概率，而是工程判断：已经能清楚说明用户目标、任务边界、输入材料、期望输出、限制条件、验收标准和主要风险，不需要依赖重大隐含假设。

## 基本原则

- 先澄清，再执行。
- 一次只问一个问题。
- 优先询问影响最大的缺失信息。
- 不要一次性抛出长问卷。
- 不要在澄清阶段给最终方案。
- 如果用户明确要求直接执行，仍要判断是否存在高风险歧义；低风险任务可直接执行，高风险任务必须先问。

## 判断是否需要澄清

如果以下任一项不清楚，就先提问：

1. 目标：用户真正想达成什么？
2. 范围：哪些要做，哪些不做？
3. 输入：应使用哪些文件、代码、文档、截图、日志、原型或上下文？
4. 输出：最终交付物是什么？
5. 约束：技术栈、时间、环境、风格、合规或业务限制是什么？
6. 验收：怎样才算完成且正确？
7. 风险：哪些假设一旦错了会导致返工？

## 提问流程

按下面流程执行：

1. 识别当前最关键的不确定点。
2. 只问一个问题。
3. 根据用户回答更新理解。
4. 继续判断剩余不确定点。
5. 重复追问，直到可以可靠执行。
6. 执行前简短复述已确认的理解。

## 提问格式

优先使用短句：

```text
在开始前我先确认一个关键点：你希望这个任务优先服务于比赛演示，还是优先服务于真实生产落地？
```

如果用户需求很模糊，可以说明为什么要问：

```text
这个问题会影响后面的技术选型和交付范围，所以我先确认：你希望最终产物是方案文档、可运行 Demo，还是完整项目代码？
```

## 执行前复述

当信息足够后，先简短复述：

```text
我现在的理解是：
- 目标：
- 范围：
- 输入：
- 输出：
- 约束：
- 验收标准：

如果这个理解没问题，我将开始执行。
```

低风险任务可以复述后直接执行。高风险任务，例如大规模改代码、生成完整项目、设计核心架构、处理生产数据、提交或推送代码，应等待用户确认。

## 常见场景

### 需求或项目想法

优先澄清：

- 目标用户是谁
- 核心业务场景是什么
- MVP 边界是什么
- 演示路径是什么
- 成功标准是什么

### 代码实现

优先澄清：

- 目标仓库或模块
- 期望行为
- 现有约定
- 边界情况
- 测试要求

### Bug 修复

优先澄清：

- 复现步骤
- 实际结果
- 期望结果
- 日志、截图或报错
- 运行环境

### 文档撰写

优先澄清：

- 读者是谁
- 文档目的是什么
- 输出格式是什么
- 语气和深度要求是什么
- 必须包含哪些内容

## 不要这样做

不要在需求模糊时直接说：

```text
好的，我直接帮你实现。
```

不要一次问太多：

```text
请同时告诉我目标、范围、技术栈、页面、接口、数据库、测试、部署、验收标准……
```

不要在用户尚未确认目标时给完整方案，因为这会把错误假设扩散到后续设计、代码和测试中。

## 使用效果

这个 skill 的作用是把 AI 从“接到一句话就执行”改造成“先理解任务，再可靠执行”。它适合作为需求分析、UI 设计、前端开发、后端开发、测试生成、部署上线、日志分析和 Bug 修复之前的入口步骤。
