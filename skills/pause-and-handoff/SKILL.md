# 暂停与任务交接

## 自动触发

只要用户语义上要求暂停、保存进度或移交任务，就直接使用本 skill，不要求出现 `pause-and-handoff`、`使用该 skill` 或其他固定口令。若用户只是询问当前状态但没有暂停意图，则正常报告状态，不生成交接文档。

## 目标

立即收束当前工作，把仅存在于会话上下文中的信息转换为可验证、可恢复、可被其他 AI 工具理解的 Markdown 交接文档。完成交接后停止继续实现。

## 执行原则

1. 以用户最新的暂停指令为最高优先级，不再开始新的功能、重构、下载、部署或长时间测试。
2. 允许执行必要且短暂的只读检查，用于确认文件、差异、测试结果和同步状态。
3. 对本代理启动且仍在运行的测试、构建或开发服务，先获取最后输出，再安全结束；不要停止用户原本就在运行的进程。
4. 明确区分“已修改”“已运行”“已验证通过”“待验证”，不能根据代码存在推断功能已经完成。
5. 记录事实和证据，不依赖“前面提到的”“刚才那个文件”等会话指代。
6. 使用普通 Markdown、标准命令和真实路径，保证 Codex、Claude及其他 AI 工具都能读取。
7. 不在交接文档中写入密码、Token、密钥、Cookie、完整身份证号或其他敏感信息。只记录变量名、安全配置位置或“需由用户提供”。

## 工作流

### 1. 立即收束

- 停止扩展任务范围。
- 检查是否有本代理启动的运行中命令或会话。
- 对已中断或可能只执行了一部分的命令，记录“可能部分完成”，不要猜测最终状态。
- 如果用户要求立即停止且不允许任何操作，直接根据已有上下文生成交接内容。

### 2. 建立事实快照

根据项目情况执行最少量检查：

- 确认当前工作目录、主要项目根目录和最终交付目录。
- 使用 `git status --short`、`git diff --stat` 和相关文件的 `git diff` 查看修改；没有 Git 时使用 `rg --files`、`cmp`、校验和或目录比较。
- 核对新增、修改、删除及未跟踪文件。
- 核对最后一次测试、构建、部署和联调的真实结果。
- 核对工作区、目标目录、远程分支或部署环境之间是否已经同步。
- 记录运行服务、端口、数据库、容器和 GUI 工具状态，但不要为了交接启动新的外部环境。

只引用本轮实际看到的证据。无法确认时标记为“未知”或“待重新验证”。

### 3. 确定交接文件位置

遵循以下顺序：

1. 用户指定路径时使用该路径。
2. 项目已有交接约定时遵循现有约定。
3. 默认在主要项目根目录创建或更新 `HANDOFF.md`。
4. 需要保留历史快照时，创建 `docs/handoffs/HANDOFF-YYYYMMDD-HHMM.md`，并让 `HANDOFF.md` 指向最新快照。
5. 存在多个项目或工作区时，把文档写入当前完整版本所在的主项目，并明确其他目录的同步状态。

优先更新已有交接文档，避免生成多个相互冲突的“最新状态”。

### 4. 编写交接文档

交接文档至少包含以下结构；不适用的章节写“无”或省略，但不能省略未完成事项和恢复步骤。

```markdown
# 任务暂停与恢复交接

更新时间：
暂停原因：用户主动暂停 / 额度限制 / 切换工具 / 其他
当前结论：已完成 / 部分完成 / 被阻塞 / 待验证

## 1. 恢复入口
- 当前完整版本路径
- 最终目标目录或仓库
- 当前分支/提交（如果有）
- 哪个目录是唯一事实来源

## 2. 原始目标与范围
- 用户最终要解决的问题
- 已确认的核心诉求
- 明确不做的内容
- 验收标准

## 3. 关键思考与决策
- 采用了什么方案
- 为什么这样选择
- 被否决的方案及原因（仅记录会影响后续工作的内容）
- 关键业务规则、数据结构和接口约定

## 4. 修改内容
| 文件/目录 | 修改内容 | 当前状态 |
|---|---|---|
| path | 具体变化 | 已验证/待验证/可能部分完成 |

## 5. 已完成事项
- 已真正完成且有证据的工作

## 6. 验证证据
| 命令/检查 | 结果 | 证据位置 |
|---|---|---|
- 测试数量、失败数、构建退出码
- 截图、报告、日志路径
- 未运行的测试必须明确写出

## 7. 未完成事项
- 按优先级列出
- 说明每项完成条件
- 区分未开始、进行中、阻塞和待验证

## 8. 阻塞、风险和已知问题
- 错误信息或阻塞原因
- 可能只完成一部分的操作
- 外部依赖、权限、网络、账号或环境问题
- 数据安全和兼容性风险

## 9. 环境与外部状态
- 运行时、依赖、端口、数据库、容器
- 服务是否仍在运行
- 本地、目标目录、远程或部署环境的同步情况
- 不记录任何秘密值

## 10. 恢复执行顺序
1. 第一步
2. 第二步
3. 第三步

## 11. 可直接执行的命令
```bash
# 使用真实工作目录和参数；敏感值使用环境变量
```

## 12. 给下一个 AI 的恢复提示
```text
请先完整阅读本交接文档，以指定目录为当前事实来源，
先核对差异和运行状态，再从“未完成事项”的第一项继续。
不要重复已经有证据证明完成的工作，也不要把待验证项写成已完成。
```
```

### 5. 检查交接质量

提交前逐项确认：

- 下一个 AI 不读取聊天记录也能理解目标和当前状态。
- 所有路径明确，必要时使用绝对路径。
- 每个关键修改都能定位到文件或目录。
- 关键决策包含理由，而不只是结论。
- 已完成事项都有测试、构建、日志、截图或人工确认等证据。
- 未运行、失败或受限的验证被明确记录。
- 中断命令、部分同步和可能产生的外部状态被明确记录。
- 恢复顺序可执行，并避免重复劳动。
- 没有秘密值和不必要的个人敏感信息。
- 文档中的“当前结论”与证据一致。

### 6. 完成交接后停止

- 返回交接文档的可点击路径。
- 用简短文字说明文档记录了什么，以及最重要的未完成项。
- 不继续编码、测试、部署或同步，除非用户随后明确要求恢复。

## 状态词规范

统一使用以下状态，避免含糊表达：

- `已完成`：实现和所需验证均已完成。
- `已修改，待验证`：代码或文件已经修改，但尚未得到足够验证。
- `进行中`：工作只完成了一部分，可以明确指出断点。
- `未开始`：尚未实施。
- `被阻塞`：因权限、环境、外部服务或必要输入无法继续。
- `可能部分完成`：命令被中断，无法确认是否全部生效。

## 特殊情况

### 工作区不是 Git 仓库

不要因此放弃交接。使用文件列表、修改时间、`cmp`、目录比较和现有日志建立事实快照，并说明缺少 Git 历史。

### 无法写入项目目录

选择当前可写且与任务最接近的目录创建交接文档，并在最终回复中说明原目标路径和实际保存路径。若完全无法写文件，在回复中输出完整 Markdown，不要只给摘要。

### 多个 AI 或多个目录交替工作

明确标出：

- 哪个目录包含最新完整修改。
- 哪些文件已经同步。
- 哪些文件尚未同步或可能冲突。
- 恢复时应先比较哪些路径。

### 测试或构建仍在运行

先等待短时间获取结果；若用户要求立即暂停，则安全终止本代理启动的进程，并将命令、最后输出、日志位置和“未完成”状态写入文档。
