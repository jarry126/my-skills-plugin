# PRD 转验收标准

## 目标

把自然语言需求转换为可判断成功或失败、可驱动 UI、前端、后端和测试实现的项目基线。不要把“按需求文档完成”当作验收标准，也不要根据实现结果反向降低标准。

## 工作流

### 1. 读取事实来源

- 定位 PRD、原型、参考项目、接口文档、业务规则和用户补充说明。
- 区分明确事实、合理推断和未知信息，标出冲突版本。
- 先理解目标用户、核心问题、主要业务闭环和交付场景，不立即编码。

### 2. 建立范围契约

将需求划分为：

- `P0`：核心闭环不可缺少，不通过则不能交付。
- `P1`：重要能力，不阻断第一版核心闭环。
- `P2`：体验优化或后续增强。
- `OUT`：本期明确不做、依赖暂缺或与核心目标关联弱。

每项必须记录划分理由。不要仅根据 PRD 篇幅判断优先级，要根据业务目标、依赖关系、风险和演示闭环判断。

### 3. 处理歧义

- 列出会改变范围、架构、数据模型、用户流程或验收结果的关键歧义。
- 能从材料和代码中验证的先自行验证。
- 无法验证且影响重大的，按执行前澄清方式一次只问一个问题。
- 低影响细节可采用保守默认值，但必须记录假设。

### 4. 生成行为验收矩阵

每条可验收需求至少包含：

| 字段 | 要求 |
|---|---|
| ID | 稳定且唯一，例如 `ORDER-001` |
| 优先级 | P0/P1/P2 |
| 业务目标 | 说明用户价值，不只描述控件 |
| 前置条件 | 用户、权限、状态和数据准备 |
| 操作 | 用户或系统执行的动作 |
| UI 结果 | 页面状态、反馈和可达操作 |
| 接口结果 | 状态码、业务码和关键字段 |
| 数据结果 | 数据库新增、修改、不变或唯一性约束 |
| 负向场景 | 越权、重复、非法状态、边界输入等 |
| 验收方式 | 单测、集成测试、API、数据库、UI 自动化、Computer Use 或人工 |
| 证据 | 日志、报告、截图、数据查询或构建产物 |

P0 规则至少包含一个正向场景和一个关键负向场景。只检查页面文案不能证明业务完成；涉及持久化时必须检查接口和数据结果。

### 5. 建立证据契约

根据项目实际选择并明确 PASS 条件：

- 编译和构建成功。
- 单元测试、集成测试及关键反例通过。
- HTTP 主链路和错误链路符合契约。
- 数据库迁移、约束和状态断言正确。
- 核心 UI 可达、无空白页和阻断性重叠。
- 脚本难以覆盖的交互使用 Computer Use 或人工补充。
- 运行日志无未解释的严重错误。
- 报告能够定位命令、结果和证据文件。

明确区分 `PASS`、`PARTIAL_PASS`、`BLOCKED` 和 `FAIL`，不得把“已实现”写成“已验证”。

### 6. 输出和确认

项目存在文档目录时，优先生成：

```text
docs/MVP范围说明.md
docs/验收标准矩阵.md
acceptance/acceptance-rules.yml
```

若项目已有命名和目录规范，遵循现有约定。高风险新项目在开始核心架构或大规模编码前，先向用户展示以下摘要并等待确认：

```text
核心目标
P0范围
明确不做
关键假设
阻断性歧义
最终验收条件
```

确认后，验收基线成为设计、开发、测试和修复的共同判断依据。后续变更必须记录原因和影响，不允许为了让测试通过而删除失败规则。

## 质量检查

提交验收基线前确认：

- 每个 P0 都能被明确判定成功或失败。
- 核心链路包含前置状态、主操作、结果和数据回看。
- 权限、重复提交、非法状态和边界输入已有反例。
- UI、接口和数据库标准没有互相矛盾。
- AppID、支付、外部账号、硬件等环境限制已标记，不会阻断可替代的 MVP 主链路。
- 每个结论都有预定证据，不依赖 AI 自我声明完成。

需要更完整的矩阵和 YAML 示例时，读取 `references/acceptance-template.md`。
